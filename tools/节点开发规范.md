# BNOS 节点开发规范

> 本文档定义 BNOS 节点的开发标准，包括目录结构、配置文件、监听器、启动脚本等。
> 如需开发**新语言的节点生成器**，请参阅 [节点生成器开发准则](节点生成器开发准则.md)。
> 如需了解 `config.json` 输入端口与参数控件的详细配置，请参阅 [config_json 开发规范](config_json_开发规范.md)。

---

## 一、节点目录结构

每个节点在 `项目/nodes/` 下必须有如下结构：

```
nodes/
└── node_{lang}_{name}/
    ├── config.json          # 节点配置（必需，见下节）
    ├── start.bat            # Windows 启动脚本（必需）
    ├── start.sh             # Linux/macOS 启动脚本（必需）
    ├── listener.{ext}       # 监听器入口（必需，持续运行）
    ├── main.{ext}           # 处理逻辑入口（必需，核心业务代码）
    ├── output.json          # 节点输出文件（运行时生成）
    ├── requirements.txt     # 依赖声明（Python 等语言必需）
    ├── .pid                 # 进程 PID 文件（启动后自动生成）
    ├── .gitignore           # 忽略编译产物/日志
    └── logs/                # 运行日志目录（自动创建）
        └── listener.log
```

### Python 节点示例

```
nodes/
└── node_python_demo/
    ├── config.json
    ├── start.bat
    ├── start.sh
    ├── listener.py
    ├── main.py
    ├── packet.py            # 数据包结构定义（可选）
    ├── output.json
    ├── requirements.txt
    ├── venv/                # 独立虚拟环境（首次启动自动创建）
    ├── .gitignore
    └── logs/
        └── listener.log
```

### Rust 节点示例

```
nodes/
└── node_rust_demo/
    ├── config.json
    ├── start.bat
    ├── start.sh
    ├── Cargo.toml           # Rust 项目配置
    ├── output.json
    ├── src/
    │   ├── main.rs          # 主处理逻辑
    │   ├── listener.rs      # 监听器
    │   └── packet.rs        # 数据包结构（可选）
    ├── .gitignore
    └── logs/
        └── listener.log
```

## 二、config.json 基础规范

每个节点必须包含 `config.json`，字段如下：

```json
{
  "node_name": "node_python_1",
  "listen_upper_file": "../data/upper_data.json",
  "output_file": "./output.json",
  "output_type": "",
  "filter": {},
  "out_connections": {},
  "port_mappings": {},
  "parameters": [],
  "input_ports": [],
  "output_ports": []
}
```

### 核心字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `node_name` | string | **必须与节点目录名一致** |
| `listen_upper_file` | string | 监听的上游数据文件路径 |
| `output_file` | string | 本节点输出文件路径 |
| `filter` | object | 注意力过滤规则（空 `{}` 则处理所有任务） |
| `output_type` | string | 输出数据类型标识 |

### 注意力过滤机制

- `filter` 为空 `{}`：节点处理**所有**上游任务（广泛注意力）
- `filter` 有键值对：节点只处理**匹配所有条件**的任务（聚焦注意力）

```json
{
  "filter": { "type": "data_task", "priority": "high" }
}
```

> **详细配置**（`parameters` / `input_ports` / `output_ports` 11 种控件类型）见：[config_json 开发规范](config_json_开发规范.md)

## 三、监听器（listener）开发规范

监听器是节点的核心入口，**必须持续运行**，不能主动退出。

### 最低功能要求

1. 读取本节点 `config.json` 获取 `listen_upper_file` 路径
2. 监听上游节点的 `output.json`（或 `upper_data.json`）文件变化
3. 根据 `filter` 规则判断是否处理该任务（注意力过滤）
4. 检查 `_processed_<node_name>` 标记，避免重复处理
5. 调用 `main` 处理逻辑，传入任务数据
6. 将处理结果写入本节点 `output.json`
7. 在上游文件中添加 `_processed_<node_name>` 处理标记
8. 日志写入 `logs/listener.log`

### 自愈机制

- 输出文件路径不存在时自动创建目录
- 上游文件不存在时等待并重试
- 异常时不崩溃，记录日志后继续监听
- 检测运行环境是否完整，缺失则自动修复

### Python 监听器伪代码

```python
import json, time, os, subprocess, sys
from datetime import datetime

def log(msg, level="INFO"):
    timestamp = datetime.now().strftime("%Y-%m-%d %H:%M:%S")
    log_dir = os.path.join(os.path.dirname(__file__), "logs")
    os.makedirs(log_dir, exist_ok=True)
    with open(os.path.join(log_dir, "listener.log"), "a", encoding="utf-8") as f:
        f.write(f"[{timestamp}] [{level}] {msg}\n")

def load_config():
    config_path = os.path.join(os.path.dirname(__file__), "config.json")
    with open(config_path, "r", encoding="utf-8") as f:
        return json.load(f)

def listen():
    config = load_config()
    node_name = config["node_name"]
    upper_file = config["listen_upper_file"]
    output_file = config["output_file"]
    filter_rules = config.get("filter", {})
    processed_key = f"_processed_{node_name}"
    
    log(f"监听器启动，监听: {upper_file}")
    
    while True:
        try:
            if not os.path.exists(upper_file):
                time.sleep(1)
                continue
            
            with open(upper_file, "r", encoding="utf-8") as f:
                data = json.load(f)
            
            # 注意力过滤
            if filter_rules:
                if not all(data.get(k) == v for k, v in filter_rules.items()):
                    time.sleep(1)
                    continue
            
            # 防重复处理
            if data.get(processed_key):
                time.sleep(1)
                continue
            
            log(f"检测到匹配任务，开始处理")
            
            # 调用 main.py 处理
            result = subprocess.run(
                [sys.executable, "main.py", json.dumps(data)],
                capture_output=True, text=True, timeout=300
            )
            
            if result.returncode == 0:
                output = json.loads(result.stdout)
                os.makedirs(os.path.dirname(output_file), exist_ok=True)
                with open(output_file, "w", encoding="utf-8") as f:
                    json.dump(output, f, ensure_ascii=False, indent=2)
                
                # 标记已处理
                data[processed_key] = True
                with open(upper_file, "w", encoding="utf-8") as f:
                    json.dump(data, f, ensure_ascii=False, indent=2)
                
                log("处理完成")
            else:
                log(f"处理失败: {result.stderr}", "ERROR")
        
        except Exception as e:
            log(f"异常: {e}", "ERROR")
        
        time.sleep(1)

if __name__ == "__main__":
    listen()
```

## 四、处理逻辑（main）开发规范

`main` 是节点的业务逻辑入口，负责**单次**处理任务并返回结果。

### 接口规范

- **输入**：接收 JSON 字符串（从命令行参数传入）或从 stdin 读取
- **输出**：输出 JSON 字符串到 stdout

### Python 示例

```python
import json, sys

def process(data: dict) -> dict:
    """
    处理输入数据，返回结果。
    
    Args:
        data: 上游传入的 JSON 数据
    
    Returns:
        dict: 处理结果，必须包含处理后的数据
    """
    input_value = data.get("data", {})
    
    # ═══ 在此实现你的业务逻辑 ═══
    result_value = input_value  # 替换为实际处理逻辑
    # ══════════════════════════
    
    return {
        "code": 0,
        "data": result_value
    }

if __name__ == "__main__":
    if len(sys.argv) > 1:
        data = json.loads(sys.argv[1])
    else:
        data = json.loads(sys.stdin.read())
    
    result = process(data)
    print(json.dumps(result, ensure_ascii=False))
```

### Rust 示例

```rust
use serde_json::Value;

fn process(data: &Value) -> Option<Value> {
    let input = data.get("data")?;
    
    // ═══ 在此实现你的业务逻辑 ═══
    let result = input.clone();
    // ══════════════════════════
    
    Some(serde_json::json!({
        "code": 0,
        "data": result
    }))
}

fn main() {
    let args: Vec<String> = std::env::args().collect();
    let data: Value = if args.len() > 1 {
        serde_json::from_str(&args[1]).unwrap()
    } else {
        serde_json::from_reader(std::io::stdin()).unwrap()
    };
    
    if let Some(result) = process(&data) {
        println!("{}", serde_json::to_string(&result).unwrap());
    }
}
```

### 输出格式

```json
{
  "code": 0,
  "data": "<处理结果>"
}
```

- `code`: `0` 表示成功，非 `0` 表示失败
- `data`: 处理结果，可以是任意 JSON 类型

## 五、启动脚本规范

### start.bat（Windows）

```bat
@echo off
setlocal enabledelayedexpansion
if not "%1"=="--no-pause" (
    cls
    chcp 65001 >nul
    echo ======================================
    echo        BNOS Node Starter
    echo ======================================
    echo.
)
cd /d "%~dp0"

REM === 环境自愈（按需实现） ===
REM 检测运行时依赖，缺失则自动修复

REM === 后台启动 ===
if not "%1"=="--no-pause" (
    echo 后台启动监听程序...
    start /b "" <启动命令>
) else (
    start /b "" <启动命令> >nul 2>&1
)

REM === 写入 PID ===
timeout /t 1 /nobreak >nul
REM 用 tasklist 获取进程 PID
for /f "tokens=2" %%i in ('tasklist /fi "imagename eq <进程名>" /nh 2^>nul') do echo %%i > .pid

if not "%1"=="--no-pause" (
    echo 监听程序已在后台运行
    pause
)
```

### start.sh（Linux/macOS）

```bash
#!/bin/bash
cd "$(dirname "$0")"
NO_PAUSE=false
[ "$1" = "--no-pause" ] && NO_PAUSE=true

# === 环境自愈（按需实现） ===

# === 后台启动 + PID ===
nohup <启动命令> > /dev/null 2>&1 &
echo $! > .pid
[ "$NO_PAUSE" = false ] && echo "监听程序已在后台运行 (PID: $(cat .pid))" && read -p "按回车键退出..."
```

### 启动脚本关键规范

| 规范 | 说明 |
|------|------|
| 支持 `--no-pause` | GUI 调用时必须静默运行，不 `pause`/`read` |
| 环境自愈 | 检测依赖缺失时自动修复（安装 venv、编译等） |
| 后台启动 | Windows 用 `start /b`，Linux/macOS 用 `nohup &` |
| PID 记录 | 启动后写入 `.pid` 文件，内容为进程 PID（一行一个数字） |
| `.pid` 格式 | 文件只写一行一个数字，如 `12345` |

## 六、节点间通信协议

### 任务数据格式

上游节点通过监听文件（`listen_upper_file` 指向的 JSON）传递任务：

```json
{
  "type": "data_task",
  "data": { "key": "value" },
  "_processed_node_python_1": true
}
```

### 输出数据格式

节点处理完成后写入 `output_file`：

```json
{
  "code": 0,
  "type": "data_result",
  "data": { "processed_key": "processed_value" }
}
```

### 防重复处理

每个节点处理完任务后，在上游文件中写入 `_processed_<node_name>` 标记：

```json
"_processed_node_python_1": true
```

监听器在每次处理前检查此标记，若已存在则跳过。

### 多节点链式处理

```
upper_data.json
    ↓ [node_python_1] 过滤 type=data_task → 处理 → output.json
    ↓ [node_python_2] 监听 node_1 的 output → 处理 → output.json
    ↓ [node_python_3] 监听 node_2 的 output → 处理 → output.json
```

## 七、开发检查清单

新建节点时请逐项确认：

- [ ] 目录结构完整（config.json / start.bat / start.sh / listener / main）
- [ ] `config.json` 中 `node_name` 与目录名一致
- [ ] `listen_upper_file` 路径正确
- [ ] `filter` 规则配置合理（空 `{}` 或具体规则）
- [ ] 监听器持续运行，不主动退出
- [ ] 监听器有防重复处理标记
- [ ] 监听器异常不崩溃，记录日志后继续
- [ ] `main` 接收 JSON 输入，输出 JSON 到 stdout
- [ ] 输出格式符合 `{"code": 0, "data": ...}` 规范
- [ ] start.bat 支持 `--no-pause` 参数
- [ ] start.bat 后台启动 + 写入 `.pid`
- [ ] start.sh 后台启动 + `echo $! > .pid`
- [ ] `.pid` 文件内容为一行一个数字
- [ ] 日志写入 `logs/listener.log`
- [ ] `.gitignore` 排除 `venv/`、`target/`、`logs/`、`.pid`、`output.json`
