# 日志系统架构重设计

**日期：** 2026-06-13  
**状态：** 已实施

---

## 1. 问题诊断

### 1.1 原架构缺陷

| 问题 | 影响 |
|------|------|
| 单文件无限追加 | `bnos_console.log` 随时间膨胀至 4.78MB+ |
| 错误与日常日志混杂 | 排查线上问题时需要从海量 INFO 中翻找 ERROR |
| Emoji 字符 | Windows GBK 编码导致 `UnicodeEncodeError` |
| 高频 DEBUG 洪水 | 鼠标交互每秒数十条 DEBUG，71% 为噪音 |
| 无自动清理 | 旧日志永久堆积 |

### 1.2 痛点总结

> "错误也要记录"、"单文件体积越来越臃肿"

核心矛盾：既要保留完整日志用于排查，又不能让文件无限膨胀。

---

## 2. 新架构设计

### 2.1 文件分离策略

```
logs/
├── bnos.log              ← 日常日志 (INFO+, 按天轮转, 保留7天)
├── bnos.log.YYYY-MM-DD   ← 历史日常日志 (自动归档)
└── bnos_error.log        ← 错误日志 (ERROR+, 按大小轮转, 1MB×5备份)
```

| 文件 | 级别 | 轮转规则 | 保留策略 | 用途 |
|------|------|----------|----------|------|
| `bnos.log` | INFO+ | 每天午夜轮转 | 保留 7 天 | 日常排查、行为追溯 |
| `bnos_error.log` | ERROR+ | 超过 1MB 轮转 | 最多 5 个备份（6MB） | 错误排查、告警追溯 |
| 控制台 | INFO+ | 无 | 无 | 开发调试 |

### 2.2 为什么这样设计

- **错误独立存储**：`bnos_error.log` 专门捕获 ERROR/CRITICAL，不受日常日志噪音干扰，排查线上问题直接看这个文件
- **按天轮转**：每天一个文件，单日日志量可控，7 天总容量预估 < 50MB
- **按大小兜底**：异常暴涨时 1MB 触发截断 + 备份，防止单文件失控
- **启动清理**：`_cleanup_old_logs()` 每次启动删掉 7 天前的 `bnos.log.*` 文件

### 2.3 三层防臃肿机制

```
调用 logger.xxx()
     │
     ▼
┌─────────────────────┐
│ ① FrequencyFilter   │  相同消息 30s 内 >5 次 → 抑制
│    ERROR/CRITICAL   │  永不过滤，确保错误不丢失
│    永远放行          │
└────────┬────────────┘
         │ (通过)
         ▼
┌─────────────────────┐
│ ② DebugLevelManager │  高频画布模块默认静默 DEBUG
│    可运行时切换      │  set_debug_mode(True) 全开
└────────┬────────────┘
         │ (通过)
         ▼
┌─────────────────────┐
│ ③ 轮转/清理          │  bnos.log 按天 → 旧文件自动删除
│    bnos_error.log    │  按 1MB → 旧备份自动截断
│    按大小            │
└─────────────────────┘
```

## 3. 核心组件（已实现）

### 3.1 SafeStreamHandler

```python
class SafeStreamHandler(logging.StreamHandler):
    """安全控制台处理器，自动处理 Windows GBK 编码下的特殊字符"""
```

Windows 控制台使用 GBK 编码，直接输出含特殊字符的日志会抛 `UnicodeEncodeError`。该 Handler 使用 `errors='replace'` 替换无法编码的字符。

### 3.2 FrequencyFilter

```python
class FrequencyFilter(logging.Filter):
    """相同日志在 30s 窗口内超过 5 次后抑制，ERROR/CRITICAL 永远放行"""
```

- 日志键：`(级别, 消息摘要, 文件名)`
- 前 5 次正常输出
- 第 6 次输出 `[FREQ] 日志过于频繁...后续抑制` WARNING
- 第 7 次起丢弃
- **ERROR/CRITICAL 永不过滤** — 保证错误不丢失

### 3.3 DebugLevelManager

```python
class DebugLevelManager(logging.Filter):
    """模块级 DEBUG 控制器"""
```

- 默认静默模块：`canvas_view`, `canvas_layout`, `edge_item`, `node_item`, `anchor_item`, `polling_manager`
- 运行时切换：`set_debug_mode(True)` 开启全模块 DEBUG
- 可动态增删静默模块：`add_quiet_module()` / `remove_quiet_module()`

### 3.4 自动清理

```python
def _cleanup_old_logs(log_dir: Path, keep_days: int = 7):
    """启动时清理超过 keep_days 天的旧日志"""
```

每次启动扫描 `logs/bnos.log.*`，删除 7 天前的归档文件。

## 4. 同步修改

### 4.1 polling_manager.py

全局日志文件监控引用从 `bnos_console.log` / `bnos_gui.log` 更新为 `bnos.log` / `bnos_error.log`，涉及：
- `_log_cache` 字典初始化
- `_init_global_watchers()` 日志文件列表
- `_poll_global_logs()` 轮询文件列表

### 4.2 Emoji 清理

所有 `.py` 文件中 emoji 字符替换为 `[TAG]` 标签前缀，作为 `SafeStreamHandler` 之外的兜底防御。

## 6. 效果对比

| 指标 | 旧架构 | 新架构 |
|------|--------|--------|
| 单文件最大体积 | 无上限（实测 4.78MB） | < 1MB（按天 + 按大小双重约束） |
| 错误定位效率 | 需 grep 全量日志 | 直接查看 `bnos_error.log` |
| 旧日志清理 | 手动删除 | 启动自动清理 7 天前 |
| 高频日志噪音 | 无过滤 | FrequencyFilter 抑制 80%+ |
| 编码兼容性 | 存在 UnicodeEncodeError | SafeStreamHandler + 标签替换 |
| 模块 DEBUG 控制 | 无 | DebugLevelManager 按模块开关 |
