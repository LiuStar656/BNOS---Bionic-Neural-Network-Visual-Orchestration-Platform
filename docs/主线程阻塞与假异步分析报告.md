## 一、关键概念定义

### 1.1 什么是"假异步"

"假异步"指代码看似在并发执行（通过 `QTimer.singleShot`、`QApplication.processEvents`、`async def` 等手段），但实际仍**完全在主线程（GUI 线程）内执行**，一旦内部工作耗时较长，仍会阻塞用户界面。

常见假异步模式：

| 模式 | 外观 | 真相 |
|------|------|------|
| A | `QTimer.singleShot(10, lambda: do_work())` | 延迟 10ms 后，仍在主线程执行 `do_work()`；若 `do_work()` 含 IO 或循环，仍阻塞 UI |
| B | `QApplication.processEvents()` | 在一个耗时的主线程函数内，手工"挤出"一点时间让 Qt 处理重绘/输入，本质仍是串行执行，且可能导致**信号重入**（同一函数被递归调用） |
| C | 把磁盘扫描或网络请求放在与 GUI 无关的线程中，但线程内部调用了 `QWidget.*` / `QGraphicsScene.*` / `main_window.*` 等 UI API | 这会触发 `QObject: Cannot create children for a parent that is in a different thread` 警告并可能崩溃 |

### 1.2 真异步（正确模式）

把耗时工作放入 **`QThread` 子类** 或 **`QRunnable`**，通过 **`Signal`** 把结果传回主线程更新 UI。Worker 线程**绝不触碰任何 `QWidget` / `QGraphicsScene` / `main_window` 对象**。

BNOS 项目中已存在的 4 个真异步 Worker：

- `ui/core/project_load_worker.py` 的 `ProjectLoadWorker(QThread)`
- `ui/main_window/node.py` 中的 `StartNodeWorker(QThread)`
- `ui/core/node_creation_worker.py` 的 `NodeCreationWorker(QThread)`
- `ui/core/node_control_service.py` 的内部 `monitor_thread`（`moveToThread` 模式）

## 三、按严重度分级的完整修复计划

### 🔴 P0 修复（必须实施）

#### P0-1. `stop_node_process` 的 `time.sleep` 移除

**当前**：
```python
def stop_node_process(node_info):
    ...
    time.sleep(0.3)  # 给进程 300ms 响应 SIGTERM
    ...
```

**目标**：改为"异步检测"——调用 `os.kill` 或 `taskkill` 后立即返回；进程是否死亡由 `node_health` 轮询自然检测（每 2 秒一次）。如果用户希望更快的视觉反馈，可以加一个 `QTimer.singleShot(300, ...)` 在 300ms 后更新 UI，但**不阻塞**。

**文件**：`ui/core/node_process.py`

**预期收益**：停止节点不再卡顿 300ms，批量停止 10 个节点不再卡顿 3 秒。

---

#### P0-2. `canvas_host._load_project_data` 异步化

**当前**：打开/切换项目时，`CanvasHost._load_project_data()` 在主线程做 `os.listdir(nodes_dir)` + 循环 `open / json.load`。

**目标**：复用 `ProjectLoadWorker` 的输出（Worker 已扫描过同样的 nodes 目录），把 Worker 的结果直接传给 CanvasHost，避免两次扫描磁盘。

或者如果 CanvasHost 需要与 `main_window.nodes_data` 不同的字段，把这部分扫描也移到 Worker 中作为附加返回。

**文件**：`ui/core/canvas_host.py`

**预期收益**：打开/切换项目时主线程阻塞进一步降低 ~100ms。

---

#### P0-3. SplashScreen 移除 `processEvents`，改为 Worker + Signal

**当前**：`splash_screen.py` 的 `append_log(msg)` / `set_progress(percent, msg)` 内部调用 `QApplication.instance().processEvents()` 让闪屏刷新。

**目标**：
1. 把 splash 期间的实际工作（创建 AppConfig、语言初始化、图标加载等）移到一个 `SplashWorker(QThread)`
2. Worker 用 `progress_signal(int, str)` 通知主线程更新进度
3. `SplashScreen` 只做 Signal → `setText` / `setProgress`，**不调用 processEvents**
4. 这样可以彻底消除"启动工作在主线程阻塞 + 手工抽事件"的反模式

**文件**：`ui/core/splash_screen.py`

**预期收益**：消除信号重入风险，启动阶段不再出现随机崩溃可能。

---

### 🟠 P1 修复（建议实施）

#### P1-1. PollingManager 后台化

**当前**：7 个轮询任务全部在主线程执行文件 IO + `psutil` 调用。

**目标**：
1. `PollingManager` 内部持有一个常驻 `PollingWorker(QThread)`
2. Worker 内部按 tick 间隔执行轮询逻辑
3. Worker 不直接更新 UI；只在检测到"真实变化"（节点死亡、日志新增、配置变更）时 emit Signal
4. 主线程接收 Signal 后更新 UI（Toast、面板、节点状态灯等）

**文件**：`ui/core/polling_manager.py`（重构）

**预期收益**：空闲 CPU 3-5% → ≤1%，轮询期间不再有 UI 抖动。

---

#### P1-2. 外部节点挂载 / 卸载移到 Worker

**当前**：`_mount_node_async` / `_unmount_node_async` 用 `singleShot` 但仍在主线程执行文件 IO + 注册表操作。

**目标**：创建 `MountNodeWorker(QThread)` / `UnmountNodeWorker(QThread)`，把磁盘读写放到后台。

**文件**：`ui/core/external_node_manager.py`

---

#### P1-3. 画布连接操作文件写分离

**当前**：每次用户在画布上建立/删除连线，`canvas_connections.py` 最多同步执行 4 次 JSON 文件写入。

**目标**：改为"内存立即更新 + 后台 Worker 延迟合并写盘"。UI 操作只更新 `main_window.nodes_data[name]['config']`，然后给后台 `ConfigWriteWorker` 发一个"需要持久化"事件。Worker 可以用去抖 500ms 合并多次写入。

**文件**：`ui/canvas/canvas_connections.py`、`ui/canvas/canvas_batch_ops.py`

---

#### P1-4. IDE 扫描后台预热

**当前**：点击按钮时才扫描。

**目标**：启动时就触发 `IdeScanWorker(QThread)`，结果缓存到 `app_config`。

**文件**：`ui/core/ide_scanner.py`

---

## 四、防假异步的开发规范

为避免未来再次引入假异步，建议在团队内建立如下规范：

### 规范 1：磁盘 IO 的"三问"

每次写 `open()` / `os.listdir()` / `os.walk()` / `glob.glob()` 之前，自问：

1. **这段代码在哪里被调用？** → 检查调用链上是否有 `QTimer.singleShot`、按钮 clicked signal、画布回调。如果最终调用者在 GUI 线程，就不应直接做 IO。
2. **数据量多大？** → 1~2 个小 JSON 文件可以接受；循环 N 次读节点目录则必须放 Worker。
3. **是否已有 Worker 做了同样工作？** → `ProjectLoadWorker` 已经扫描 `nodes_dir`，避免重复工作。

### 规范 2：禁用 `time.sleep` / `processEvents`

在 GUI 相关代码（主线程路径上的所有函数）中：

- ❌ 禁止 `time.sleep(x)` —— 改用 `QTimer.singleShot(x*1000, ...)` 非阻塞等待
- ❌ 禁止 `QApplication.processEvents()` —— 除非有明确理由（极少场景），否则用 Worker 替代
- ⚠️ 谨慎使用 `subprocess.run(..., timeout=x)` 在主线程 —— 小命令可以，大扫描必须放 Worker

### 规范 3：Worker 内的"禁触碰"清单

任何 `QThread.run()` 内部，**绝对禁止**以下操作：

- 任何 `QWidget` / `QDialog` 的创建、修改、显示
- `QGraphicsScene` / `QGraphicsItem` 的创建或修改（画布节点 Item 必须在主线程构建）
- `main_window.*` 的属性访问或方法调用
- 任何可能触发 `paintEvent` 的操作

**允许的操作**：文件读写、JSON 解析、`subprocess`、`psutil`、纯数据结构构建、`logger.info(...)` 日志写入。

**主线程与 Worker 之间的唯一桥梁**：`pyqtSignal` / `Signal` —— Worker emit Signal，主线程在槽函数中接收并更新 UI。

### 规范 4：新增 Worker 的标准模板

在 `ui/core/` 下新增 `xxx_worker.py`，推荐结构：

```python
from PySide6.QtCore import QThread, Signal

class XxxWorker(QThread):
    """说明：这个 Worker 做什么工作，输入/输出什么数据"""
    progress = Signal(int, str)         # (percent, message)
    finished = Signal(dict, list)        # 成功：返回工作结果
    failed = Signal(str)                 # 失败：返回错误信息

    def __init__(self, param1, param2, parent=None):
        super().__init__(parent)
        # 只存纯数据参数，不存 UI 对象引用

    def run(self):
        """纯数据操作，绝不触碰 UI"""
        try:
            # ... 耗时工作 ...
            self.progress.emit(50, "half done")
            # ... 继续工作 ...
            self.finished.emit(result_dict, result_list)
        except Exception as e:
            import traceback
            traceback.print_exc()
            self.failed.emit(str(e))
```

## 七、总结摘要

| 指标 | 数值 |
|------|------|
| QTimer.singleShot 假异步 | 13 处（6 处有阻塞风险） |
| 主线程 open/json 文件 IO | ≥22 处 |
| `processEvents` 手工抽事件 | 2 处 |
| `time.sleep` 显式阻塞 | 至少 1 处（stop_node_process） |
| 已有真异步 Worker | 4 个（架构方向正确，但覆盖不足） |

**核心结论**：BNOS 的架构方向是正确的（关键路径已用 `QThread`），但在"次要但高频"的轮询系统、画布加载、节点停止、外部节点挂载、连接配置写盘这几条路径上仍是同步或假异步。这正是用户感受到"打开项目/切换画布明显卡顿"的根源。

按本文档 P0/P1 实施后，主线程将从"1500ms 阻塞"下降到"300ms 以内"，用户体感从"明显卡"变为"流畅"。
