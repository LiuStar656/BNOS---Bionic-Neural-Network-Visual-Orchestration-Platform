# BNOS 卡顿问题迭代优化方案

**文档版本**：v1.0
**编制日期**：2026-06-16
**适用版本**：BNOS Console 主分支（Python 3.14 + PySide6 6.11）
**文档性质**：技术方案 + 实施记录

---

## 一、问题定义

### 1.1 卡顿场景概述

在 BNOS 节点编排平台的日常使用中，以下场景存在明显的用户感知卡顿：

| 场景 | 触发方式 | 现象描述 | 感知阻塞时间（50节点项目基准） |
|------|---------|---------|------------------------------|
| 打开项目 | 菜单 "文件 → 打开项目" | Toast 提示后窗口整体冻结，画布内容突然弹出 | 800ms ~ 5s |
| 加载画布布局 | 打开项目后自动读取 `canvas_layout.json` | 画布空白几秒后节点/连线突然全部出现 | 300ms ~ 3s |
| 切换项目画布 | CanvasHost 顶部 Tab 切换到另一个项目 | 切换瞬间窗口冻结 | 200ms ~ 2s |
| 平移/缩放画布 | 中键拖动 / Ctrl + 滚轮 | 大项目中有明显掉帧感 | 视觉层面 |
| 批量操作 | 批量启动/停止/删除节点 | 操作期间 UI 无响应 | 随节点数线性增长 |

### 1.2 问题分层分析

通过代码走查，卡顿根源可分为三层：

**（A）渲染层**：每次 QGraphicsItem 变更触发 `FullViewportUpdate`，导致整个画布视口重绘。节点 Item 未启用 `DeviceCoordinateCache`，滚动/缩放时逐帧重新绘制。

**（B）主线程 IO 层**：项目加载与节点刷新的磁盘扫描、JSON 解析全部在 GUI 线程同步执行。50 节点规模时，单轮 `os.listdir + open + json.load` 循环可消耗 100ms+。

**（C）算法与数据层**：`load_layout` 对节点数据做 2~3 次冗余遍历；切换项目时重复调用磁盘扫描（数据已在内存中）；轮询系统所有任务均在主线程执行文件 IO。

## 二、目标与验收标准

### 2.1 量化目标（以 50 节点 / 100 连线的中等项目为基准）

| 指标 | 优化前 | 目标 |
|------|--------|------|
| 打开项目主线程阻塞 | ~1500ms | ≤400ms |
| 加载画布主线程阻塞 | ~1200ms | ≤300ms |
| 切换画布主线程阻塞 | ~800ms | ≤150ms |
| 空闲场景平移/缩放帧率 | 20-30 fps | 50+ fps |
| 空闲 CPU 占用（无节点运行） | 3-5% | ≤1% |

### 2.2 功能零退化约束

- 节点拖放、连线、锚点、选中等交互行为完全不变
- Toast / 面板 / 右键菜单功能完全不变
- `canvas_layout.json` 读写格式完全向后兼容
- 退出程序时配置持久化机制保持不变

## 三、完整实施方案

### 阶段 1：渲染层优化（已实施 ✅）

**目标**：不触碰业务逻辑，仅调整 Qt 渲染 API 与缓存模式，降低每次重绘开销。

#### 改动 1.1 — 视口更新模式 Full → Smart

| 项目 | 内容 |
|------|------|
| 文件 | `ui/canvas/canvas_view.py` |
| 改动 | `self.setViewportUpdateMode(QGraphicsView.ViewportUpdateMode.FullViewportUpdate)` → `SmartViewportUpdate` |
| 原理 | Qt 的 `SmartViewportUpdate` 内部计算 dirty region，仅重绘受影响区域。BNOS 中节点彼此独立，此模式效果显著 |
| 验证 | 拖动一个节点，观察其他节点不闪烁 |

#### 改动 1.2 — NodeItem 启用设备坐标缓存

| 项目 | 内容 |
|------|------|
| 文件 | `ui/canvas/items/node_item.py` |
| 改动 | `self.setCacheMode(QGraphicsItem.CacheMode.NoCache)` → `DeviceCoordinateCache`（初始化处 + `_destroy_detailed` / `_ensure_rect` 中禁用缓存后必须重新启用） |
| 原理 | 静态节点被缓存为离屏像素块，平移时 Qt 直接 blit，无需重新执行 `paint()` |
| 关键注意 | 节点尺寸变化（`setRect`）时必须先设 `NoCache` 再 `setRect` 再恢复 `DeviceCoordinateCache` + `update()`，否则旧缓存会被复用 |

#### 改动 1.3 — 收敛全场景刷新为局部刷新

| 项目 | 内容 |
|------|------|
| 文件 | `ui/canvas/items/node_item.py` |
| 改动 | `if self.scene(): self.scene().update()` → `self.update()` |
| 原理 | `scene().update()` 标记整个场景 dirty，触发整张画布重绘；`self.update()` 只标记本节点 boundingRect 为 dirty |

**阶段 1 预期收益**：拖动/缩放帧率 2~3× 提升，无功能风险。

### 阶段 2：IO 异步化（已实施 ✅）

**目标**：把磁盘扫描和 JSON 解析从 GUI 线程剥离，主线程只负责接收结果并更新 UI。

#### 改动 2.1 — 新增 ProjectLoadWorker

| 项目 | 内容 |
|------|------|
| 文件 | `ui/core/project_load_worker.py`（新增） |
| 父类 | `QThread` |
| 信号 | `progress(int, str)` — 进度百分比 + 描述<br>`finished(dict, list, list)` — `(nodes_data, mounted_nodes, running_nodes)`<br>`failed(str)` — 失败描述 |
| `run()` 核心流程 | ① `os.listdir(nodes_dir)` ② 循环读取 `config.json` 组装 `nodes_data` ③ `NodeRegistry.load/sync_from_scan/save` 同步注册表 ④ 处理挂载节点 ⑤ `detect_running_nodes` 检测后台进程 ⑥ emit `finished` |
| 关键约束 | **绝对不在 Worker 内触碰任何 UI 对象**（main_window、QGraphicsScene、QTreeWidget 等）。所有数据返回主线程后再应用 |

#### 改动 2.2 — project_open / project_refresh 走 Worker 模式

| 项目 | 内容 |
|------|------|
| 文件 | `ui/core/project_manager.py` |
| `project_open` 新流程 | ① 路径校验 ② Toast 显示"正在打开项目" ③ 主线程清空 `main_window.nodes_data`/`connections` ④ 创建 `ProjectLoadWorker` ⑤ 绑定信号 `progress→Toast更新`、`finished→_apply_after_refresh`、`failed→Toast错误` ⑥ `worker.start()` 启动线程 |
| `project_refresh` 新流程 | 同样走 Worker，与上面一致（去掉了旧的 `async_mode` 参数分支 — 因为现在始终异步） |
| `_apply_after_refresh(main_window, running_nodes)` 新函数 | 在主线程执行：① 把 Worker 返回的 `nodes_data` 赋给 `main_window.nodes_data` ② 同步面板 `update_node_list` ③ `sync_all_nodes_display` 到画布 ④ 更新 running 节点状态 ⑤ Toast 完成提示 |
| 删除的旧代码 | 整个 `_project_refresh_async()` 函数（内部 `for item in os.listdir(nodes_dir)` + 同步 `registry` + `QTimer.singleShot(10, update_ui)`） |

**阶段 2 预期收益**：打开项目 GUI 不再冻结，主线程仅在最后 50ms 有一次批量赋值。

### 阶段 3：算法与数据层优化（已实施 ✅）

#### 改动 3.1 — 画布切换不再重新扫描磁盘

| 项目 | 内容 |
|------|------|
| 文件 | `ui/main_window/__main__.py` |
| 触发函数 | `_on_canvas_changed(new_canvas)` |
| 改动前 | 调用 `project_refresh()`（重扫描磁盘） + `_refresh_panels()` |
| 改动后 | 调用 `sync_canvas_data_to_main_window(new_canvas)`（用内存中已有数据） → `_refresh_panels()` 直接更新面板 → 去掉 `project_refresh()` 调用 |
| 说明 | `sync_canvas_data_to_main_window` 已同步新画布的 `nodes_data`，无需再次读磁盘 |

#### 改动 3.2 — load_layout 合并为单次双遍历

| 项目 | 内容 |
|------|------|
| 文件 | `ui/canvas/canvas_layout.py` |
| 旧逻辑 | ① 遍历 layout_nodes 更新已存在节点位置 + ② 再次遍历补建缺失节点（甚至第三次遍历兜底） + ③ 遍历 edges 创建连线 + ④ 锚点校验 + ⑤ config 反推（再次遍历 nodes） + ⑥ 最后整体 `for edge in edges: edge.update_path()` 强制刷新 |
| 新逻辑 | ① 单次遍历 layout_nodes：存在则更新位置/样式/颜色，不存在则收集到 `missing_nodes` ② `missing_nodes` 一次性批量创建 ③ edges 单次遍历创建连线 ④ config 反推保留但走 node-by-ref 映射 ⑤ 移除末尾冗余的 `update_path()` 循环（每条边创建时已显式调用） |
| 额外优化 | 引入 `node_by_ref = {node: name for name, node in nodes.items()}` 映射，把 edge 构建中的 `for n, nd in nodes.items(): if nd == e.start_node: sn = n` 内层遍历改为 O(1) 查找 |

#### 改动 3.3 — 面板更新增量化（已验证价值，保留为可选后补）

`update_node_list` 每次 `clear()` + 重建整树，在多节点场景下有明显开销。此改动逻辑较复杂且与面板组件结构耦合深，作为**阶段 3 可选项**，不纳入本次强制实施。

### 阶段 4：轮询降噪（待实施 🔄）

#### 改动 4.1 — 主定时器频率降倍

| 项目 | 内容 |
|------|------|
| 文件 | `ui/core/polling_manager.py` |
| 改动 | `self._timer.setInterval(1000)` → `setInterval(self.tick_duration * 1000)`，其中 `self.tick_duration = 2`（秒） |
| 说明 | 主 tick 从每秒一次改为每 2 秒一次，整体轮询密度减半 |

#### 改动 4.2 — 任务间隔差异化

| 任务 | 原间隔 | 新间隔 | 说明 |
|------|--------|--------|------|
| `node_health` | 2 秒 | 2 秒（1 tick） | 进程生死检测保持高频 |
| `global_logs` | 2 秒 | 4 秒（2 tick） | 全局日志更新可降频 |
| `global_config` | 2 秒 | 4 秒（2 tick） | 全局配置更新可降频 |
| `node_logs` | 2 秒 | 4 秒（2 tick） | 节点日志更新可降频 |
| `node_config` | 2 秒 | 6 秒（3 tick） | 配置文件变更不频繁 |
| `node_output` | 2 秒 | 6 秒（3 tick） | 输出文件同上 |
| `app_state` | 5 秒 | 10 秒（5 tick） | 应用状态低频即可 |

#### 改动 4.3 — 后台化（进阶，见下一阶段 Top5 #1）

把 PollingManager 内部 7 个任务全部迁移到常驻 QThread Worker，只通过 Signal 把"变更事件"抛回主线程。这是更大的改动，在第 4+ 阶段进行。

## 四、Top 5 深度优化（待实施）

本部分为进一步的深度优化项，是本方案接下来的实施目标。

### 4.1 PollingManager 后台化

**问题**：当前 7 个轮询任务全部在主线程执行 `open`/`read`/`psutil.pid_exists`，多节点时每数秒都会有一批文件 IO 阻塞主线程。

**方案**：把 PollingManager 改为内部一个常驻 QThread（`PollingWorker`），Worker 内部执行轮询；仅在检测到"真实变更"（日志内容变化、节点死亡、配置更新）时通过 Signal 通知主线程做 UI 更新。

**文件**：`ui/core/polling_manager.py`（重构）

**预期**：空闲 CPU 从 3-5% 降至 ≤1%，轮询期间不再有 UI 抖动。

### 4.2 canvas_host._load_project_data 异步化

**问题**：打开项目时 `CanvasHost._load_project_data()` 仍是 `os.listdir(nodes_dir) + open(json)` 主线程同步扫描，与 `ProjectLoadWorker` 做了重复工作但在主线程。

**方案**：要么让 `ProjectLoadWorker` 返回完整的"节点配置字典"给画布直接使用，要么把这段扫描也移到一个独立的 `CanvasLoadWorker`。

**文件**：`ui/core/canvas_host.py`

**预期**：打开项目阻塞时间进一步降低 ~80%。

### 4.3 移除 processEvents 和 time.sleep

**问题**：`splash_screen.py` 用 `QApplication.processEvents()` 手工抽事件来让启动画面不卡死——这暗示启动工作仍在主线程阻塞。`node_process.stop_node_process()` 用 `time.sleep(0.3)` 等待进程死亡。两者都是"阻塞 + 手工挤出事件"的反模式，会导致信号重入与随机崩溃。

**方案**：
- Splash 阶段改为 QThread Worker + Signal 更新进度，彻底移除 `processEvents()`
- `stop_node_process` 改为 `QTimer.singleShot(300, check_again)` 轮询检测进程死亡，不阻塞事件循环

**文件**：`ui/core/splash_screen.py`、`ui/core/node_process.py`

### 4.4 连接操作文件写分离

**问题**：`canvas_connections.py` 每次用户做连接/删除操作时，最多执行 4 次 `open(config.json, 'w') + json.dump`，全在主线程。

**方案**：改为"内存立即更新 + 后台 Worker 批量合并写盘"。UI 操作只更新 `main_window.nodes_data[name]['config']`，然后触发一个"配置变更"事件给后台 `ConfigWriteWorker` 合并写入。

**文件**：`ui/canvas/canvas_connections.py`、`ui/canvas/canvas_batch_ops.py`

**预期**：用户在画布上的连线/删除操作响应即时。

### 4.5 IDE 扫描后台预热

**问题**：点击"在 IDE 打开"按钮时才执行 `subprocess.run(powershell, timeout=5)` 扫描 VS Code/IDE 路径，5 秒超时时间主线程完全阻塞。

**方案**：在程序启动初期（splash 阶段或启动后）触发一个 `IdeScanWorker(QThread)`，把找到的 IDE 路径缓存到 `app_config`。用户点击按钮时只做一次内存查表。

**文件**：`ui/core/ide_scanner.py`

## 七、风险与回滚

| 风险 | 描述 | 严重度 | 缓解措施 |
|------|------|--------|---------|
| R-1 | SmartViewportUpdate 在复杂场景产生撕裂 | 中 | 实施后立即测试；异常则退回 `MinimalViewportUpdate` |
| R-2 | DeviceCoordinateCache 在节点内容变化后缓存未失效 | 中 | 每次 `setRect`/样式变化后必须 `self.setCacheMode(NoCache) → setRect → setCacheMode(DeviceCoordinateCache) → update()` |
| R-3 | Worker 线程内意外调用 UI API → 崩溃 | 高 | Code Review 检查 Worker 不触碰任何 `QWidget`/`QGraphicsScene`/`main_window`；在 Worker 开头加线程身份断言 |
| R-4 | `load_layout` 重构引入回归（连线/锚点不正确） | 中 | 保留旧函数作为 `load_layout_v1` 参考；新增专项测试检查 20+ 节点/40+ 连线项目 |
| R-5 | 去掉 `project_refresh` 后节点列表面板有脏数据 | 中 | `sync_canvas_data_to_main_window` 必须覆盖 `nodes_data`，再让 `update_node_list` 重新构建树 |
