# BNOS 下一步发展计划

## 📊 项目现状综合分析

### 项目定位
**BNOS** (Bionic Neural Network Program Operating System) 是一个基于 PySide6 的**桌面可视化节点编排平台**，类似 VueFlow/Node-RED 的工作流引擎。

### 核心架构（已完成）

| 模块 | 状态 | 核心职责 |
|------|------|----------|
| **启动层** | ✅ 完成 | `launcher.py` → `bnos_console.py`，带进度条的闪屏启动 |
| **主窗口** | ✅ 完成 | `BNOSMainWindow`，无边框设计，PS式布局 |
| **画布系统** | ✅ 完成 | `NodeCanvas`，组合模式（10个组合层），VueFlow风格无限画布 |
| **进程管理** | ✅ 基础完成 | `ProcessManager` + `node_process.py`，子进程生命周期管理 |
| **轮询管理** | ✅ 完成 | `PollingManager`，后台线程文件IO，多任务调度 |
| **事件总线** | ✅ 完成 | `EventBus`，发布-订阅解耦 |
| **历史管理** | ⚠️ 部分完成 | `HistoryManager`，支持撤销/重做/跳转，但 `load_history` 未实现 |
| **动作系统** | ✅ 完成 | `ActionRegistry` + `ActionFactory`，命令模式 |
| **面板系统** | ✅ 完成 | `PanelManager`，节点列表/资源监控/节点监控 |
| **CanvasHost** | ✅ 完成 | 多画布管理，空白缓冲层，独立数据隔离 |

### 2. 节点启动逻辑重复
**文件**: `ui/core/node_control_service.py` vs `ui/core/node_process.py`

**问题**：存在两套独立的节点启动/停止逻辑：
- `NodeControlService`: 检测文件类型启动 (`main.py`, `index.js`, `Cargo.toml`)
- `node_process.py`: 使用 `start.json` 配置启动

**后果**：代码重复、行为不一致、维护困难。

### 3. 历史记录加载未实现
**文件**: `ui/core/commands/history_manager.py`


**问题**：项目关闭后历史记录丢失，用户体验差。

### 4. 缺少统一的配置管理
**问题**：颜色值、尺寸等硬编码在多个文件中：

`ui/canvas/canvas_view.py`

**文件**: `run_tests.py` 存在但内容未知，核心模块缺乏单元测试。

## 🚀 下一步发展方向（优先级排序）

### 优先级 1：核心功能补全

#### 1.1 实现核心进程 IPC 命令分发
**目标**: 让 `core_process.py` 真正工作起来

**实现方案**:
```python
def _on_message(self, msg):
    action = msg.get("action")
    params = msg.get("params", {})
    
    if action == "start_node":
        self._handle_start_node(params)
    elif action == "stop_node":
        self._handle_stop_node(params)
    elif action == "refresh_project":
        self._handle_refresh_project(params)
    # ... 更多命令
```

**价值**: 实现主进程与核心业务进程的真正分离，提高稳定性和可扩展性。

---

#### 1.2 统一节点启动逻辑
**目标**: 合并 `NodeControlService` 和 `node_process.py` 的启动逻辑

**方案**: 以 `node_process.py` 的 `start.json` 方案为基础（更灵活），废弃 `NodeControlService` 中的启动逻辑。

**价值**: 消除代码重复，保证行为一致性。

---

#### 1.3 实现历史记录持久化
**目标**: 完成 `load_history` 方法

**实现**: 反序列化 `history.json`，重建 `Command` 对象

**价值**: 项目重启后保留用户操作历史，提升体验。

---

### 优先级 2：性能优化

#### 2.1 画布渲染优化
**问题**: 当前 `NodeCanvas` 使用 QGraphicsView，节点数量多时卡顿

**方案**:
- 引入虚拟滚动/视口裁剪
- 优化 `paint()` 方法，减少重绘区域
- 增加节点渲染缓存策略

---

#### 2.2 轮询频率动态调整
**问题**: `PollingManager` 使用固定 2 秒间隔，空闲时浪费 CPU

**方案**:
- 空闲时降低轮询频率（如 10 秒）
- 活动时提高频率（如 1 秒）
- 基于系统负载动态调整

---

### 优先级 3：架构重构

#### 3.1 引入状态管理模式
**问题**: 状态分散在 `main_window`, `canvas`, `canvas_host` 中，同步复杂

**方案**: 引入集中式状态管理（类似 Vuex/Redux），统一数据流

---

#### 3.2 统一配置系统
**目标**: 将分散的硬编码配置集中管理

**方案**: 扩展 `AppConfig`，支持主题、布局、快捷键等配置的统一存取

---

### 优先级 4：新功能扩展

#### 4.1 节点调试支持
**功能**: 为节点进程添加调试端口、日志断点、实时变量查看

**价值**: 开发者友好，提高调试效率

---

#### 4.2 性能分析面板
**功能**: 实时显示节点 CPU/内存/网络使用情况，支持性能火焰图

**价值**: 运维友好，便于性能优化

---

#### 4.3 节点模板系统
**功能**: 支持创建节点模板，快速生成同类节点

**价值**: 提高开发效率

---

#### 4.4 协作功能（远期）
**功能**: 多用户实时协作编辑画布

**方案**: 基于 WebSocket 的状态同步
