## 2. 主要问题分析

### 2.1 NodeCanvas 对 parent_window 的强依赖

**问题：** NodeCanvas 中有 **30 处**调用 `self.parent_window`，子进程中 `parent=None` 会导致功能失效。

**依赖项清单：**

```python
# 数据访问
self.parent_window.nodes_data              # 节点数据
self.parent_window.current_project_path   # 项目路径

# 方法调用
self.parent_window.start_selected_node_by_name()    # 启动节点
self.parent_window.stop_selected_node_by_name()     # 停止节点
self.parent_window.export_node()                    # 导出节点
self.parent_window._force_stop_all_nodes()          # 批量停止
self.parent_window.open_color_settings()            # 颜色设置
self.parent_window.open_settings()                  # 设置对话框

# UI 交互
self.parent_window.show_toast()                     # 显示通知
self.parent_window.refresh_nodes()                  # 刷新节点
self.parent_window.clear_connections()              # 清空连线
```

**影响范围：** `add_node_to_canvas()`、`remove_node_from_canvas()`、右键菜单功能等几乎所有功能。

---

### 2.2 API 签名不匹配

**canvas_process.py 中的调用：**
```python
self.canvas.add_node_to_canvas(node_name, info)  # 2个参数
```

**当前实际方法签名：**
```python
def add_node_to_canvas(self, node_name):  # 只有1个参数
    # 内部依赖 self.parent_window.nodes_data[node_name]
```

**问题：** 参数数量不匹配，且方法内部逻辑已改变。

---

### 2.3 架构变更的影响

| 架构变更 | 对子进程的影响 | 严重程度 |
|---------|---------------|---------|
| **CanvasHost** | 画布现在在 Dock 中，不是独立窗口 | 🔴 高 |
| **主窗口 Mixin 拆分** | 增加了子进程需要复制的复杂度 | 🟡 中 |
| **ApplicationContext** | 单例模式在多进程中难以共享 | 🔴 高 |
| **EventBus** | 事件系统在进程间无法直接传递 | 🔴 高 |
| **DIContainer** | 依赖注入容器在子进程中为空 | 🟡 中 |
| **ShutdownOrchestrator** | 关闭编排无法跨进程协调 | 🟡 中 |
| **PollingManager** | 轮询管理器状态无法同步 | 🟡 中 |

## 4. 修复建议

### 4.1 短期方案（推荐）

**建议：** 暂时保持进程隔离功能禁用

**操作：**
- 维持 `CANVAS_PROCESS_MODE = False`
- 标记进程隔离相关代码为"原型阶段"
- 专注于完善单进程架构的稳定性

**优点：**
- 风险低，不影响现有功能
- 可以继续优化其他优先级更高的功能

---

### 4.2 长期方案（如需启用）

如果未来确实需要进程隔离功能，需要完成以下工作：

#### 阶段 1：重构 NodeCanvas

- [ ] 移除对 `self.parent_window` 的直接依赖
- [ ] 引入回调接口或事件系统
- [ ] 将数据访问和方法调用解耦

#### 阶段 2：实现完整的 IPC 消息处理

- [ ] 完成 `panel_process.py` 的 `_on_message()`
- [ ] 完成 `core_process.py` 的 `_on_message()`
- [ ] 完善 `canvas_process.py` 的消息处理
- [ ] 实现所有 IPC Action 常量的处理逻辑

#### 阶段 3：建立跨进程状态同步机制

- [ ] 设计 ApplicationContext 的跨进程同步方案
- [ ] 实现 EventBus 的跨进程事件传递
- [ ] 建立进程间数据一致性保障

#### 阶段 4：重新设计 CanvasHost 替代方案

- [ ] 设计子进程中的画布宿主架构
- [ ] 或完全移除 CanvasHost 依赖，回到独立窗口模式

#### 阶段 5：集成测试

- [ ] 编写进程隔离的集成测试
- [ ] 测试崩溃恢复机制
- [ ] 测试 IPC 通信稳定性

---

### 4.3 替代方案

考虑使用以下替代架构：

1. **多线程而非多进程**
   - 避免 IPC 复杂性
   - 共享内存更容易
   - 但需要注意 GIL 限制

2. **使用共享内存 + 消息队列**
   - 更快的进程间通信
   - 更复杂的同步逻辑

3. **保持单进程，优化性能**
   - 使用 Profiler 找出瓶颈
   - 针对性优化热点代码
   - 使用 C 扩展重写性能关键部分
