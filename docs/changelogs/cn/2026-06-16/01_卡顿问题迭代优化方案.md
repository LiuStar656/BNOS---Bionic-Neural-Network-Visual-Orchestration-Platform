# 📊 BNOS 卡顿问题迭代优化方案

**方案版本**: v1.0
**编制日期**: 2026-06-16
**问题范围**: 打开项目卡顿 / 加载画布卡顿 / 多项目切换卡顿
**预期收益**: 打开/切换场景的主线程阻塞时间降低 70%+
**风险等级**: 低 ~ 中（渲染层改动为 Qt 标准API切换，线程化改动有约定俗成的安全模式）

---

## 一、问题定义与现状分析

### 1.1 卡顿三大场景

| 场景 | 触发方式 | 当前表现 | 感知阻塞时间 |
|------|---------|---------|-------------|
| 打开项目 | 菜单 "打开项目" → 选择项目目录 | Toast 卡死 → 窗口"未响应" → 画布内容一次性弹出 | 500ms ~ 5s（随节点数增长） |
| 加载画布 | 打开项目后自动加载 `canvas_layout.json` | 画布空白几秒 → 节点/连线突然全部出现 | 300ms ~ 3s |
| 切换项目 | 点击 CanvasHost 顶部 Tab，切换到另一个画布 | 切换瞬间窗口冻结 → 面板全部重建 | 200ms ~ 2s |

### 1.2 性能瓶颈分层诊断

通过代码走查，识别出如下瓶颈路径与热点函数：

#### （A）渲染层瓶颈

| 瓶颈编号 | 位置 | 当前实现 | 问题 |
|---------|------|---------|------|
| A-1 | `ui/canvas/canvas_view.py:75` | `setViewportUpdateMode(FullViewportUpdate)` | 任意 Item `update()` 都会触发整个 viewport 重绘，放大 N 倍开销 |
| A-2 | `ui/canvas/items/node_item.py:36` | `setCacheMode(NoCache)` | 节点 item 不使用 Qt 离屏缓存，滚动/缩放时每个节点重新 paint |
| A-3 | `ui/canvas/items/node_item.py:275` | `if self.scene(): self.scene().update()` | 每个节点 `apply_style` 都触发整场景更新，N 个节点 = N 次全场景刷新 |

#### （B）主线程 IO 瓶颈

| 瓶颈编号 | 位置 | 当前实现 | 问题 |
|---------|------|---------|------|
| B-1 | `ui/core/project_manager.py:167-208` `_project_refresh_async` | `for item in os.listdir(nodes_dir): ... with open(config_path)... json.load(f)` | 同步扫描目录 + N 次同步磁盘读 + N 次 JSON 解析，全部在 GUI 线程执行 |
| B-2 | `ui/canvas/canvas_layout.py:117` `load_layout` | `with open(layout_file) ... json.load` | 同步读取 + 解析 `canvas_layout.json`，大型项目可达几 MB |
| B-3 | `ui/core/app_config.py:74` + 所有 `app_config.save()` 调用 | 同步写配置文件，无 batch | 每次 toggle 面板/终端都写磁盘，叠加时放大 UI 抖动 |

#### （C）数据结构与算法瓶颈

| 瓶颈编号 | 位置 | 当前实现 | 问题 |
|---------|------|---------|------|
| C-1 | `ui/canvas/canvas_layout.py:98-444` `load_layout` | 节点遍历 2 次 + 连线遍历 3 次 + config 兜底再遍历 1 次 | 总复杂度 O(N+E) 被放大到 O(3N+4E) |
| C-2 | `ui/panels/node_list_panel.py:114` `update_node_list` | `self.node_tree.clear()` → 重新逐个 `addTopLevelItem` | 每次调用整棵树重建，3 个面板（panel/floating/dock）同时重建 = 3 倍开销 |
| C-3 | `ui/main_window/__main__.py:285-313` `_on_canvas_changed` | `sync_canvas_data_to_main_window` → `refresh_nodes` → `_refresh_panels` | 切换画布时重复扫描磁盘（数据已在内存） |

#### （D）后台轮询噪声

| 瓶颈编号 | 位置 | 当前实现 | 问题 |
|---------|------|---------|------|
| D-1 | `ui/core/polling_manager.py:84` | 1000ms 主定时器 + 6 个子任务并行 | 每秒固定触发 1 次主循环，空闲时也有 CPU 噪声 |
| D-2 | `ui/core/polling_manager.py:282-318` | `_poll_node_config` / `_poll_node_output` 每次都读文件内容 | 即使文件未变化也执行读操作，节点数多了会有累积 IO |

## 二、目标与验收标准

### 2.1 量化目标（以 50 节点、100 连线的中等项目为基准）

| 指标 | 当前 | 目标 | 判定方法 |
|------|------|------|---------|
| 打开项目主线程阻塞 | ~1500ms | ≤400ms | `cProfile` / Qt 事件分析器，或手工 `time.time()` 在关键路径打点 |
| 加载画布主线程阻塞 | ~1200ms | ≤300ms | 同上 |
| 切换画布主线程阻塞 | ~800ms | ≤150ms | 同上 |
| 空闲场景平移/缩放帧率 | 20~30 fps | 50+ fps | Qt 自带 FPS 手动测算 |
| 空闲 CPU 占用（无节点运行） | 3~5% | ≤1% | Task Manager |

### 2.2 功能零退化约束

- 节点拖放、连线、锚点、选中等交互行为完全不变
- Toast/面板/右键菜单的功能完全不变
- `canvas_layout.json` 读写格式完全向后兼容（已保存项目无需重写）
- `app_config.json` 格式兼容

## 三、详细迭代方案（3 轮迭代）

### 迭代 1：渲染层 — 最小改动、最大收益（优先级 P0）

**目标**: 不触碰业务逻辑，仅调整 Qt 渲染 API，降低刷新开销。

#### 任务 1.1：视口更新模式切换

| 项目 | 内容 |
|------|------|
| 文件 | `ui/canvas/canvas_view.py` |
| 当前行 | 第 75 行 `self.setViewportUpdateMode(QGraphicsView.ViewportUpdateMode.FullViewportUpdate)` |
| 变更 | 改为 `QGraphicsView.ViewportUpdateMode.SmartViewportUpdate` |
| 说明 | Qt 的 SmartViewportUpdate 会自动判定需要重绘的区域。BNOS 场景里节点彼此独立，不会出现"多个相邻 item 过度裁剪"的问题 |
| 验证 | 拖动一个节点，观察其他节点是否不闪烁。放大缩小时整体画面是否流畅 |

#### 任务 1.2：NodeItem 启用像素缓存

| 项目 | 内容 |
|------|------|
| 文件 | `ui/canvas/items/node_item.py` |
| 当前行 | 第 36 行 `self.setCacheMode(QGraphicsItem.CacheMode.NoCache)`；第 222、257、286 行多次显式设回 `NoCache` |
| 变更 | 统一改为 `DeviceCoordinateCache`；在 `setRect` / `_style.apply` 之后显式调用 `self.update()` 触发缓存失效 |
| 注意事项 | 使用 `DeviceCoordinateCache` 的约束：**变换（缩放/旋转）时缓存自动失效**，BNOS 场景正好适用。节点尺寸变化时必须手动 `update()` 触发重新缓存 |
| 验证 | 静态场景放大缩小无明显马赛克；拖动时 FPS 提升 |

#### 任务 1.3：移除/收敛 `scene().update()` 全量刷新

| 项目 | 内容 |
|------|------|
| 文件 | `ui/canvas/items/node_item.py` |
| 当前行 | `apply_style` / `apply_status` 末尾调用 `self.scene().update()` |
| 变更 | 改为 `self.update()`，只更新该节点自身 |
| 说明 | `scene().update()` 会标记整个场景为 dirty；`self.update()` 只标记该 item 的 boundingRect。由于 NodeItem 本身有合理的 boundingRect，足以覆盖状态变化 |

**迭代 1 预期收益**：滚动/缩放/拖动 → 3~5× 更快，卡顿显著降低。

---

### 迭代 2：IO 异步化 — 把磁盘/CPU 密集操作从主线程剥离（优先级 P0）

**目标**: 打开项目时 GUI 不冻结，Toast 动画正常播放。

#### 任务 2.1：新增 `ProjectLoadWorker(QThread)`

| 项目 | 内容 |
|------|------|
| 新增文件 | `ui/core/project_load_worker.py` |
| 输入参数 | `project_path: str` |
| 输出数据 | `nodes_data: dict`（节点名 → `{config, path, status, ...}`）<br>`mounted_nodes: list[dict]`（从 `NodeRegistry` 恢复的挂载节点）<br>`error_message: str` |
| 信号定义 | `progress = Signal(int, str)` — 进度百分比 + 当前操作<br>`finished = Signal(dict, list)` — 成功时发出<br>`failed = Signal(str)` — 失败时发出 |
| 核心流程 | `run()` 方法内：<br>① `os.listdir(nodes_dir)`<br>② 循环读每个 `config.json` 并 `json.load`<br>③ 调用 `NodeRegistry.sync_from_scan()`（只读注册表操作，无 UI 依赖）<br>④ 调用 `detect_running_nodes()`<br>⑤ 组装结果 `nodes_data` / `mounted_nodes`，emit `finished` |
| 线程安全 | **绝对不在线程内触碰任何 Qt UI 对象**（包括 `main_window`、`QGraphicsScene`、`QTreeWidget` 等）。所有 UI 更新通过 `Signal` 回到主线程 |
| 进度提示 | 每处理 5 个节点 emit 一次 `progress`，避免信号风暴 |

#### 任务 2.2：`project_open` 改造为 Worker + Signal 模式

| 项目 | 内容 |
|------|------|
| 文件 | `ui/core/project_manager.py` |
| 当前实现 | `QTimer.singleShot(10, _open_project_async)`，内部同步执行 `project_refresh` |
| 新流程 | ```
project_open(main_window, project_dir):
  1. 路径/有效性校验（仍在主线程，轻量）
  2. main_window.show_toast("正在打开项目...", "info")
  3. worker = ProjectLoadWorker(project_dir)
  4. worker.progress.connect(lambda pct, msg: 更新 Toast 文案)
  5. worker.finished.connect(_on_load_finished)   # 回到主线程
  6. worker.failed.connect(lambda err: main_window.show_toast(err, "error"))
  7. worker.start()

_on_load_finished(nodes_data, mounted_nodes):
  1. main_window.nodes_data = nodes_data         # 纯数据赋值
  2. 处理挂载节点（纯数据写入）
  3. main_window._canvas_host.add_canvas_dock(...) # 创建 NodeCanvas
  4. main_window._canvas_host.<canvas>.load_layout()  # 见迭代 3
  5. QTimer.singleShot(300, lambda: restore_canvas_host_state())
  6. main_window.show_toast("已打开项目", "success")
``` |
| 关键约束 | Worker 内部**绝不**访问 `main_window`、**绝不**调用任何 `QGraphics` API。`add_canvas_dock` 必须在主线程的 `_on_load_finished` 中执行 |

#### 任务 2.3：`project_refresh` 也走 Worker 复用

| 项目 | 内容 |
|------|------|
| 变更 | `project_refresh(main_window, async_mode=True)` 复用同一个 `ProjectLoadWorker`。`async_mode=False`（用于打开项目）改成与 `async_mode=True` 同样异步执行 |
| 说明 | 这一步会让 `F5` 刷新节点列表时不再冻结 UI |

**迭代 2 预期收益**：打开项目从"冻结 N 秒"变为"Toast 平滑播放 + 后台加载"，主线程仅在最后 50~100ms 有一次批量赋值。

### 迭代 3：算法与数据层优化（优先级 P1）

**目标**: 消除重复工作、降低 O(N) 扫描次数、切换项目时避免重新扫描磁盘。

#### 任务 3.1：`load_layout` 合并为单次双遍历

| 项目 | 内容 |
|------|------|
| 文件 | `ui/canvas/canvas_layout.py` |
| 当前实现 | 第 1 遍：更新已有节点位置/样式 → 第 2 遍：补建缺失节点 → 第 1 遍：连线 → 第 1 遍：anchor binding 校验 → 第 1 遍：config 推断（又读全部 config）→ 第 1 遍：update_path |
| 优化后 | ```
load_layout():
  1. 读 layout_file / 颜色设置 (1 次 IO)
  2. 1 次遍历 layout_data["nodes"]:
     - 节点在 self.nodes 中: 更新 pos / rect / style / 颜色
     - 节点不在: 暂存到 missing_nodes 列表
  3. 批量创建 missing_nodes 的 NodeItem（1 次批量构造）
  4. 1 次遍历 layout_data["edges"]: 创建 EdgeItem + 锚点绑定
  5. 1 次整体 anchor 绑定校验（遍历 edges 1 次）
  6. 仅当 layout 中有缺失节点时，执行 config 推断（按需触发）
  7. 1 次整体 update_path（edges 集体更新）
  8. 恢复 view state（滚动条/中心坐标）
``` |
| 效果 | `O(3N+4E)` → `O(N+2E)`，对大型项目节省 40~50% 布局加载时间 |

#### 任务 3.2：画布切换时跳过重新扫描

| 项目 | 内容 |
|------|------|
| 文件 | `ui/main_window/__main__.py` |
| 触发函数 | `_on_canvas_changed(new_canvas)` |
| 当前 | `sync_canvas_data_to_main_window` → `refresh_nodes` → `_refresh_panels`（3 个面板整树重建） |
| 新流程 | `sync_canvas_data_to_main_window` 已同步了 `nodes_data`，**不调用 `refresh_nodes`**（避免重新扫描磁盘），直接 `_refresh_panels`（后续还会继续优化） |

#### 任务 3.3：`update_node_list` 增量更新（可选，先做 3.1/3.2 验证）

| 项目 | 内容 |
|------|------|
| 文件 | `ui/panels/node_list_panel.py`, `ui/panels/node_list_dock.py` |
| 思路 | 不 `clear()` 整棵树，改为：<br>① 收集现有 item 的节点名集合<br>② 新 nodes_data 的节点名集合<br>③ 差异运算：删除旧的、新增缺的、更新已有的状态文本<br>④ 保留用户的展开/折叠状态 |
| 影响 | 切换项目时面板刷新从 ~150ms 降至 ~20ms |
| 注意 | 此改动涉及较多逻辑，推荐在 3.1 / 3.2 验证通过后再做 |

#### 任务 3.4：`app_config.save()` 防抖 batch

| 项目 | 内容 |
|------|------|
| 文件 | `ui/core/app_config.py` |
| 变更 | `save()` 内部加 500ms `QTimer.singleShot` 防抖。连续多次 `save()` 只写一次磁盘 |
| 注意 | 必须保证**程序关闭前**写一次，即 `ShutdownOrchestrator` 需要显式调用 `app_config.flush()` |

### 迭代 4：后台轮询降噪（优先级 P2）

**目标**: 空闲时 CPU 占用降低，事件循环不被"习惯性任务"淹没。

#### 任务 4.1：PollingManager 主定时器降频 + 分层

| 项目 | 内容 |
|------|------|
| 文件 | `ui/core/polling_manager.py` |
| 主定时器 | 1000ms → 2000ms |
| 子任务间隔 | 进程检测 4s、节点日志检测 4s、节点配置检测 6s、全局日志 5s、应用状态检测 10s |
| 优化要点 | 每个子任务都加上"last_mtime 没变则不读文件内容"的早期退出（当前代码已有，只需确保全面生效） |

## 五、风险评估与缓解措施

| 风险 | 描述 | 影响 | 缓解措施 |
|------|------|------|---------|
| R-1 | `SmartViewportUpdate` 在某些场景下产生撕裂/抖动 | 视觉 | 实施后立即测试拖动、框选、缩放操作。如有异常，退回 `MinimalViewportUpdate`（更保守的选项） |
| R-2 | `DeviceCoordinateCache` 节点内容变化后缓存未刷新 | 视觉 | 每次 `setRect` / `apply_style` / `apply_status` 后强制 `self.update()` 让 Qt 重建缓存 |
| R-3 | QThread 中意外调用 UI API → 崩溃或 Qt 警告 | 稳定 | Code Review 严格检查 `ProjectLoadWorker.run()` 不触碰任何 `QWidget`/`QGraphicsItem`/`main_window`。使用 `QThread.currentThread()` 与 `app.thread()` 断言辅助校验 |
| R-4 | `load_layout` 重构可能引入回归（连线/锚点绑定不正确） | 功能 | 保留原函数为 `load_layout_v1`（带 `# DEPRECATED` 注释但不删除），先跑原函数验证通过后再切到新实现。新增专用测试：打开含 20+ 节点/40+ 连线的项目，检查每条连线两端是否绑定到正确锚点 |
| R-5 | 切换项目去掉 `refresh_nodes` 后，新节点列表面板可能有脏数据 | 功能 | `sync_canvas_data_to_main_window` 已用新画布的数据覆盖 `main_window.nodes_data`，面板仅需 `update_node_list` 用新数据重建。确保新节点数据完整无误后再切换 |
| R-6 | `app_config.save()` 防抖导致"刚 toggle 然后立刻关闭"时数据丢失 | 数据 | 加 `flush()` 方法，`ShutdownOrchestrator` 关闭前调用一次。内部保留"dirty"标记，关闭时强制落盘 |

## 七、回滚方案

每一轮迭代都是**独立可回滚**的原子变更：

| 迭代 | 回滚方式 |
|------|---------|
| 迭代 1 | 把 `SmartViewportUpdate` 改回 `FullViewportUpdate`；把 `DeviceCoordinateCache` 改回 `NoCache`；把 `self.update()` 改回 `self.scene().update()` |
| 迭代 2 | 保留 `ProjectLoadWorker` 类（可以留着不用），把 `project_open` 改回原 `QTimer.singleShot + _open_project_async` 调用链 |
| 迭代 3 | `load_layout` 保留 v1 实现函数作为 fallback，出问题时直接把入口改回调用 `load_layout_v1`；其他项目级改动可单独 revert |
| 迭代 4 | 调回 PollingManager 的 1000ms 定时器即可 |

## 八、后续观察与扩展方向

本轮优化后，以下方向可作为下一阶段的待办：

1. **节点创建分批化（分帧加载）**：`load_layout` 中即使合并了遍历，创建 50+ 个 `NodeItem` 仍然必须在主线程执行。可以进一步改造为"每帧创建 N 个"，让 UI 在加载过程中仍保持响应（可选，需评估收益）
2. **EdgeItem 也启用缓存**：线条 item 可以用 `NoCache` 以外的模式，但因线条频繁更新路径，需要谨慎评估
3. **节点列表虚拟化**：如未来项目规模突破 200 节点，需考虑 `QTreeWidget` → `QListView` + 自定义 Model 的虚拟化重构
4. **Layout 文件格式优化**：当前 `canvas_layout.json` 对大型项目可能几 MB，可按需引入增量写入 / 二进制格式
