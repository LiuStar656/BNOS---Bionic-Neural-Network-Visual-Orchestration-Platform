## 🔧 Phase 1：修复节点配置菜单启动后自动关闭问题

### 1.1 问题分析

**问题现象**：在节点配置菜单中点击"启动节点"按钮后，配置菜单会自动关闭，用户需要重新打开菜单才能进行后续操作（如查看状态、停止节点等）。

**用户痛点**：
- 启动节点后想立即查看运行状态，但菜单已关闭
- 需要频繁启动/停止节点时，每次都要重新打开菜单
- 调试节点时需要在菜单和监控面板之间来回切换

### 1.2 功能需求

修改节点配置菜单的行为：启动节点后菜单保持打开状态，用户可手动关闭或继续操作。

### 1.3 任务分解

| 任务 | 描述 | 优先级 |
|------|------|--------|
| **T1.1** | 定位节点配置菜单中启动节点的代码位置 | 高 |
| **T1.2** | 移除启动成功后自动关闭菜单的逻辑 | 高 |
| **T1.3** | 添加菜单关闭按钮，支持用户手动关闭 | 中 |
| **T1.4** | 测试验证修改不影响其他菜单功能 | 中 |

### 1.4 技术实现要点

**问题代码位置**（预估）：
```python
# ui/dialogs/node_config_dialog.py 或类似文件
def on_start_node(self):
    # 启动节点逻辑
    start_node(self.node_path)
    self.close()  # ← 移除这行自动关闭逻辑
```

**修改方案**：
```python
def on_start_node(self):
    # 启动节点逻辑
    success = start_node(self.node_path)
    if success:
        # 更新菜单状态显示（如按钮状态、状态文本）
        self.update_status_display()
        # 不再自动关闭菜单
    else:
        # 启动失败时显示错误提示
        self.show_error_message()
```

### 1.5 验收标准

✅ 点击"启动节点"后，配置菜单保持打开状态
✅ 菜单中显示节点运行状态更新（如状态指示灯变化）
✅ 用户可继续在菜单中操作（停止节点、查看日志等）
✅ 用户可点击关闭按钮手动关闭菜单

## 🪟 Phase 2：修复 Windows 启动节点时弹出终端窗口问题

### 2.1 问题分析

**问题现象**：在 Windows 系统上，通过 GUI 启动节点或新建节点时，会短暂弹出终端窗口，影响用户体验。

**根本原因**：在 Windows 上启动子进程时，未使用 `CREATE_NO_WINDOW` flag 导致控制台应用程序 (`python.exe`) 启动时显示窗口。

**问题代码位置**：
1. `ui/core/node_process.py` 第232行 - 启动节点时
2. `tools/python_create_node.py` 第106、123行 - 新建节点创建虚拟环境时

**问题代码示例**：
```python
# ui/core/node_process.py
if os.name == 'nt':
    process = subprocess.Popen(
        [python_exe, listener_py],
        cwd=node_path,
        creationflags=subprocess.CREATE_NEW_PROCESS_GROUP,  # ❌ 缺少 CREATE_NO_WINDOW
        stdout=subprocess.PIPE,
        stderr=subprocess.PIPE,
        text=True
    )

# tools/python_create_node.py
result = subprocess.run(
    [sys.executable, "-m", "venv", venv_path],
    capture_output=True,
    text=True,
    timeout=120
    # ❌ 缺少 creationflags=CREATE_NO_WINDOW
)
```

### 2.2 解决方案

| 任务 | 描述 | 优先级 |
|------|------|--------|
| **T2.1** | 修改 `node_process.py`，在 Windows 上添加 `CREATE_NO_WINDOW` flag | 高 |
| **T2.2** | 修改 `python_create_node.py`，在 Windows 上添加 `CREATE_NO_WINDOW` flag | 高 |
| **T2.3** | 检查其他 `subprocess.run/Popen` 调用是否需要同样修复 | 中 |
| **T2.4** | 回归测试验证修复不影响正常功能 | 中 |

### 2.3 技术实现要点

**修复方案**：
```python

if os.name == 'nt':
    creationflags = subprocess.CREATE_NO_WINDOW
else:
    creationflags = 0

    [sys.executable, "-m", "venv", venv_path],

    text=True,
    timeout=120,
    creationflags=creationflags
```

**需要检查的其他位置**：

- `ui/panels/node_monitor.py` 第324-328行

### 2.4 验收标准

✅ Windows 上启动节点时不弹出终端窗口
✅ Windows 上新建节点时不弹出终端窗口

## 🐍 Phase 3：修复 JSON 启动无法启动虚拟环境问题

### 3.1 问题分析

**问题现象**：通过 JSON 配置启动 Python 节点时，无法正确激活虚拟环境，导致节点启动失败。

**根本原因**：`tools/python_create_node.py` 生成的 `start.json` 配置文件**缺少虚拟环境路径配置**，导致启动器无法准确定位 venv 中的 Python 解释器。

**当前生成的 start.json 结构**：
```json
{
    "nodes": [
        {
            "name": "node_python_xxx",
            "path": "...",
            "config": {
                "listen_upper_file": "../data/upper_data.json",
                "output_file": "./output.json"
            }
        }
    ]
}
```

**缺少的关键配置**：`python_exe_path`（虚拟环境 Python 解释器路径）

### 3.2 解决方案

| 任务 | 描述 | 优先级 |
|------|------|--------|
| **T3.1** | 修改 `python_create_node.py`，在 `start.json` 中添加 venv 路径配置 | 高 |
| **T3.2** | 更新 `node_process.py`，支持从 `start.json` 读取 venv 配置 | 高 |
| **T3.3** | 添加启动前 venv 有效性检查（目录完整性、Python 版本验证） | 高 |
| **T3.4** | 编写单元测试验证修复（覆盖 Windows/Linux/macOS） | 中 |
| **T3.5** | 集成测试：创建节点 → JSON 启动 → 验证运行状态 | 中 |

### 3.3 技术实现要点

**1. 修改生成器输出的 start.json 结构**：
```json
{
    "nodes": [
        {
            "name": "node_python_xxx",
            "path": "/full/path/to/node",
            "python_exe": "/full/path/to/node/venv/Scripts/python.exe",
            "config": {
                "listen_upper_file": "../data/upper_data.json",
                "output_file": "./output.json"
            }
        }
    ]
}
```

**2. 更新启动器逻辑**：
- 优先从 `start.json` 的 `python_exe` 字段读取 Python 解释器路径
- 若未配置，则回退到默认路径 `venv/Scripts/python.exe`（Windows）或 `venv/bin/python`（Linux/macOS）

**3. 验证机制**：
- 启动前检查 venv 目录结构完整性
- 验证 Python 解释器可执行性和版本兼容性
- 失败时提供明确的错误提示（如 venv 缺失、Python 版本不兼容）

### 3.4 验收标准

✅ `python_create_node.py` 生成的 `start.json` 包含 `python_exe` 字段  
✅ 通过 JSON 配置启动 Python 节点时，能正确激活虚拟环境  
✅ 启动失败时显示清晰的错误信息（如 venv 缺失、Python 版本不兼容）  
✅ 跨平台支持（Windows 10/11、Linux、macOS）  
✅ 单元测试覆盖率 ≥ 80%

```python
def attach_toolbar(self):
    """创建并挂载绘图工具栏到画布左侧（默认隐藏）"""
    if self.toolbar:
        return self.toolbar
    view = self.canvas.viewport()
    self.toolbar = DrawToolbar(view)
    self.toolbar.setParent(view)
    self.toolbar.setGeometry(0, 0, 56, view.height())
    self.toolbar.hide()  # ← 默认隐藏
    # ... 信号连接保持不变
    return self.toolbar

def toggle_toolbar(self):
    """切换工具栏显示/隐藏"""
    if self.toolbar:
        if self.toolbar.isVisible():
            self.toolbar.hide()
        else:
            self.toolbar.show()
```

**画布工具栏添加切换按钮**：
```python
# 在画布工具栏添加绘图工具切换按钮
self._draw_tool_btn = QPushButton(get_icon("pencil"))
```

**快捷键支持**：

## 📊 Phase 7：开发画布节点直接显示运行状态功能

### 7.1 功能需求

在画布上的节点本体上直接展示节点的实时运行状态信息，无需切换到监测面板，提升用户监控效率。

**适用范围**：仅应用于**方形版**和**详细版**节点，圆形版节点不显示状态信息。

### 7.2 显示内容

| 指标 | 显示方式 | 更新频率 | 适用节点形式 |
|------|----------|----------|--------------|
| **运行状态** | 状态指示灯（绿/灰/黄/红） | 实时 | 方形版、详细版 |
| **CPU 占用** | 百分比数字 + 进度条 | 每 2 秒 | 方形版、详细版 |
| **内存使用** | MB 数值 + 进度条 | 每 2 秒 | 方形版、详细版 |
| **运行时长** | HH:MM:SS 格式 | 每秒 | 方形版、详细版 |
| **状态提示** | 悬停 Tooltip | 按需 | 方形版、详细版 |

**不同节点形式的状态显示差异**：

| 节点形式 | 状态显示内容 |
|----------|-------------|
| **详细版** | 完整显示：状态指示灯 + CPU + 内存 + 运行时长 + 详细状态文本 |
| **方形版** | 精简显示：状态指示灯 + CPU 百分比 + 内存 MB 数值 |
| **圆形版** | 不显示状态信息（保持最小化显示） |

| **T7.7** | 用户体验测试与调整 | 中 |

### 7.4 技术实现要点

**性能优化策略**：
- 仅对画布视口内的节点进行监控采样
- 仅对方形版和详细版节点进行状态更新，圆形版跳过

| **连接方式** | 数据流连线（端口-端口） | 上下级文件监听（文件路径配置） |
| **视图呈现** | 画布内嵌节点控件 | 直接在画布上渲染，支持三种表现形式 |
| **表现形式** | 固定样式 | 详细版/方形版/圆形版三种样式随时切换 |
| **适用场景** | 适合数据流式处理（如图像处理） | 适合参数密集型节点（如 LLM 调用、本地推理） |
