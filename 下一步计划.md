# BNOS 进程隔离方案

> 状态：实施中 | 目标：主窗口不崩 → 画布崩不影响面板 → 面板崩不影响主程序 → 局部崩溃=局部重启

---

## 一、最终目标

**主进程永不崩溃 → 画布崩溃自动重启 → 面板崩溃自动重启 → 核心业务 7×24 稳定运行 → 软件永不闪退。**

---

## 二、进程划分（完全匹配现有架构）

### 进程 1：主进程（超级轻量）

| 负责模块 | 不负责 |

| `i18n / theme / dark_title_bar` | — |

## 四、通信方案：QLocalSocket + JSON 事件

- **Qt 原生，跨平台，无需额外依赖**
- 现有 Signal/Slot 架构直接兼容：只把信号从「内存发射」改成「跨进程发射」
- 组件代码一行不改，只加一层通信包装

---

| 步骤 | 内容 | 收益 | 状态 |
|------|------|------|------|

| **第2步** | 面板进程分离 | 左右面板不影响主窗口 | ✅ 基础设施完成 |

## 六、核心优势

| 优势 | 说明 |
|------|------|
| **不改现有架构** | 组件化、Mixin、Manager 全部保留，只是把模块装进不同进程跑 |
| **彻底解决一崩全崩** | 画布/面板/主进程/业务后台四层崩溃隔离 |
| **AI 可直接开发** | 模块边界已拆完，只需做子进程启动 + 跨进程消息 + 崩溃监控 |
| **天然跨平台** | Windows / macOS 完全通用 |

---

## 七、设计准则

1. **不改业务**：所有业务逻辑、函数签名、类结构保持不变
2. **不改文件拆分**：现有 46 个模块文件一个不动，只加进程包装层
3. **新增功能只增文件**：子进程启动器、跨进程通信基类、崩溃监控器
4. **向后兼容**：先完成单进程验证，再逐步拆分进程

## 八、第1步实施细节

### 新增文件

| 文件 | 职责 |
|------|------|
| `ui/core/ipc.py` | QLocalServer/QLocalSocket + JSON 编解码基类 |
| `ui/core/process_manager.py` | 子进程启动/停止/崩溃重启/PID 管理 |
| `ui/canvas/canvas_process.py` | 画布子进程入口，独立 QApplication + NodeCanvas |

### IPC 消息协议

```json
{
  "action": "action_name",
  "params": {},
  "request_id": "uuid"
}
```

### 第1步数据流

```
main_window                   canvas_process
    │                              │
    ├─ add_node_to_canvas()        │
    ├─ ipc.send({"add_node",...})──┼→ 接收 → canvas.add_node()
    │                              │
    ├─ canvas.node_selected        │
    │◄─ ipc.send({"selected",...})─┼─ node 被点击
    │                              │
    ├─ process_manager.monitor()   │
    │   canvas 崩溃？              │
    │   ✗ → 自动重启 canvas_process│
```
