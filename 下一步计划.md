## 🔧 Phase 1：修复 JSON 启动无法启动虚拟环境问题

### 1.1 问题分析

**问题现象**：通过 JSON 配置启动 Python 节点时，无法正确激活虚拟环境，导致节点启动失败。

**根本原因**：`tools/python_create_node.py` 生成的 `start.json` 配置文件**缺少虚拟环境路径配置**，导致启动器无法准确定位 venv 中的 Python 解释器。

**当前生成的 start.json 结构**：
```json
{
    "nodes": [
        {
            "name": "node_python_xxx",
            "path": "...",
            "config": {
                "listen_upper_file": "../data/upper_data.json",
                "output_file": "./output.json"
            }
        }
    ]
}
```

**缺少的关键配置**：`python_exe_path`（虚拟环境 Python 解释器路径）

### 1.2 解决方案

| 任务 | 描述 | 优先级 | 预估时间 |
|------|------|--------|----------|
| **T1.1** | 修改 `python_create_node.py`，在 `start.json` 中添加 venv 路径配置 | 高 | 1 天 |
| **T1.2** | 更新 `node_process.py`，支持从 `start.json` 读取 venv 配置 | 高 | 1 天 |
| **T1.3** | 添加启动前 venv 有效性检查（目录完整性、Python 版本验证） | 高 | 1 天 |
| **T1.4** | 编写单元测试验证修复（覆盖 Windows/Linux/macOS） | 中 | 1 天 |
| **T1.5** | 集成测试：创建节点 → JSON 启动 → 验证运行状态 | 中 | 1 天 |

### 1.3 技术实现要点

**1. 修改生成器输出的 start.json 结构**：
```json
{
    "nodes": [
        {
            "name": "node_python_xxx",
            "path": "/full/path/to/node",
            "python_exe": "/full/path/to/node/venv/Scripts/python.exe",
            "config": {
                "listen_upper_file": "../data/upper_data.json",
                "output_file": "./output.json"
            }
        }
    ]
}
```

**2. 更新启动器逻辑**：
- 优先从 `start.json` 的 `python_exe` 字段读取 Python 解释器路径
- 若未配置，则回退到默认路径 `venv/Scripts/python.exe`（Windows）或 `venv/bin/python`（Linux/macOS）

**3. 验证机制**：
- 启动前检查 venv 目录结构完整性
- 验证 Python 解释器可执行性和版本兼容性
- 失败时提供明确的错误提示（如 venv 缺失、Python 版本不兼容）

### 1.4 验收标准

✅ `python_create_node.py` 生成的 `start.json` 包含 `python_exe` 字段  
✅ 通过 JSON 配置启动 Python 节点时，能正确激活虚拟环境  
✅ 启动失败时显示清晰的错误信息（如 venv 缺失、Python 版本不兼容）  
✅ 跨平台支持（Windows 10/11、Linux、macOS）  
✅ 单元测试覆盖率 ≥ 80%

| **T2.3** | 修改 `rust_create_node.py`，生成包含运行时配置的 `start.json` | 高 | 2 天 |
| **T2.4** | 实现 Rust 节点启动器（编译产物检测 + 双进程启动） | 高 | 3 天 |
| **T2.5** | 实现 Rust 节点停止器（进程组管理） | 高 | 2 天 |
| **T2.6** | 集成到统一进程管理框架 `node_process.py` | 高 | 2 天 |
| **T2.7** | 添加编译产物自动检测与重建机制 | 中 | 2 天 |
| **T2.8** | 编写集成测试 | 中 | 2 天 |

### 2.3 技术实现要点

**启动流程设计**：
```
JSON 配置 → 验证 Rust 工具链 → 检查编译产物 → 启动双进程（主程序 + listener）→ 写入 PID 文件
```

**Rust 节点 start.json 结构**：
```json
{
    "nodes": [
        {
            "name": "node_rust_xxx",
            "path": "/full/path/to/node",
            "binary_path": "/full/path/to/node/target/release/node_rust_xxx",
            "listener_path": "/full/path/to/node/target/release/node_rust_xxx_listener",
            "config": { ... }
        }
    ]
}
```

**关键特性**：
- 编译产物缺失时自动触发 `cargo build --release`
- 使用进程组确保停止时彻底清理所有子进程
- 支持 Windows 和 Unix-like 系统

✅ Rust 节点可通过 JSON 配置正常启动

✅ 启动失败时自动检测并重建编译产物  
✅ 停止操作能彻底终止所有相关进程

- 调整过程流畅，不影响其他窗口操作

| 右上角 | SizeFDiagCursor |
| 左下角 | SizeFDiagCursor |
| 右下角 | SizeBDiagCursor |

**调整逻辑**：
- 记录拖拽起始位置和窗口原始尺寸

## �️ Phase 6：修复 Windows 启动节点时弹出终端窗口问题

### 6.1 问题分析

**问题现象**：在 Windows 系统上，通过 GUI 启动节点或新建节点时，会短暂弹出终端窗口，影响用户体验。

**根本原因**：在 Windows 上启动子进程时，未使用 `CREATE_NO_WINDOW` flag 导致控制台应用程序 (`python.exe`) 启动时显示窗口。

**问题代码位置**：
1. `ui/core/node_process.py` 第232行 - 启动节点时
2. `tools/python_create_node.py` 第106、123行 - 新建节点创建虚拟环境时

**问题代码示例**：
```python
# ui/core/node_process.py
if os.name == 'nt':
    process = subprocess.Popen(
        [python_exe, listener_py],
        cwd=node_path,
        creationflags=subprocess.CREATE_NEW_PROCESS_GROUP,  # ❌ 缺少 CREATE_NO_WINDOW
        stdout=subprocess.PIPE,
        stderr=subprocess.PIPE,
        text=True
    )

# tools/python_create_node.py
result = subprocess.run(
    [sys.executable, "-m", "venv", venv_path],
    capture_output=True,
    text=True,
    timeout=120
    # ❌ 缺少 creationflags=CREATE_NO_WINDOW
)
```

### 6.2 解决方案

| 任务 | 描述 | 优先级 | 预估时间 |
|------|------|--------|----------|
| **T6.1** | 修改 `node_process.py`，在 Windows 上添加 `CREATE_NO_WINDOW` flag | 高 | 0.5 天 |
| **T6.2** | 修改 `python_create_node.py`，在 Windows 上添加 `CREATE_NO_WINDOW` flag | 高 | 0.5 天 |
| **T6.3** | 检查其他 `subprocess.run/Popen` 调用是否需要同样修复 | 中 | 0.5 天 |
| **T6.4** | 回归测试验证修复不影响正常功能 | 中 | 1 天 |

### 6.3 技术实现要点

**修复方案**：
```python
import subprocess

if os.name == 'nt':
    creationflags = subprocess.CREATE_NO_WINDOW
else:
    creationflags = 0

# 启动节点时
process = subprocess.Popen(
    [python_exe, listener_py],
    cwd=node_path,
    creationflags=creationflags,
    stdout=subprocess.PIPE,
    stderr=subprocess.PIPE,
    text=True
)

# 新建节点时
result = subprocess.run(
    [sys.executable, "-m", "venv", venv_path],
    capture_output=True,
    text=True,
    timeout=120,
    creationflags=creationflags
)
```

**需要检查的其他位置**：
- `bnos_console.py` 第87行
- `ui/core/process_manager.py` 第40行
- `ui/dialogs/node_config_dialog.py` 第326、360行
- `ui/panels/node_monitor.py` 第324-328行

### 6.4 验收标准

✅ Windows 上启动节点时不弹出终端窗口  
✅ Windows 上新建节点时不弹出终端窗口  
✅ Linux/macOS 行为不受影响  
✅ 节点进程仍能正常输出日志到文件
